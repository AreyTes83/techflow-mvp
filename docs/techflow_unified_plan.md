## 1. АРХІТЕКТУРА СИСТЕМИ

### 1.1. Ролі та ієрархія управління

Система має ієрархічну структуру з правами призначення задач "зверху вниз":

```
Біг Босс (Генеральний директор)
  └── Керівник техслужби
        ├── ставить задачі → будь-яким технікам
        ├── ставить задачі → заступникам
        └── Заступники (один рівень, без вкладеності)
              └── ставлять задачі → будь-яким технікам
```

**Ієрархія — лише один рівень заступників.** Заступник заступника — неможливий. Зони відповідальності є, але поки що заступники можуть ставити задачі ВСІМ технікам без обмежень.

**Ролі:**
- **Працівник магазину** — прив'язаний до торгової точки. Створює запити (текст/голос/фото), підтверджує виконання робіт, ставить оцінки.
- **Технік** — має профіль із золотим балансом, спеціалізацією, рейтингом. Працює через Квест-борд.
- **Заступник керівника** — може ставити задачі будь-яким технікам (зони відповідальності плануються, але поки без обмежень). Бачить статуси задач.
- **Керівник техслужби** — ставить задачі технікам і заступникам, моніторить всю мережу, аналізує ефективність.
- **Біг Босс** — фінансовий контроль, бюджет, стратегічні звіти.

**Реалізація ієрархії в БД:**
```sql
-- Таблиця ролей
roles (
  id UUID PRIMARY KEY,
  name TEXT NOT NULL,           -- 'store_staff', 'technician', 'deputy', 'manager', 'bigboss'
  can_assign_to JSONB,          -- ['technician'] для deputy, ['technician','deputy'] для manager
  permissions JSONB              -- список дозволених дій
);

-- Зв'язка user_roles (many-to-many, дозволяє кілька ролей)
user_roles (
  user_id UUID,
  role_id UUID,
  assigned_by UUID               -- хто призначив цю роль
);
```

**Правило призначення задач:** Користувач може ставити задачі лише тим ролям, які вказані в `can_assign_to` його ролі.

### 1.2. Технологічний стек

| Компонент | Технологія | Зона відповідальності |
|---|---|---|
| Бот + TMA бекенд | Python (aiogram 3.x) | Telegram webhook, TMA, user interaction, state |
| Фонова оркестрація | n8n (local) | AI-класифікація, нотифікації, аналітика, звіти |
| База даних | Supabase (PostgreSQL) | Дані, Auth, Storage, Realtime |
| AI | OpenAI API | Класифікація, pricing, whisper, sentiment |
| Адмін-панель | Next.js або Vite (окремий веб-портал) | Діаграми, heatmap, звіти для Керівника та Біг Боса |
| Маршрутизація | OpenRouteService (безкоштовний API) | Оптимізація порядку точок маршруту |

**Розподіл aiogram / n8n:**
- **aiogram** = фронт-контролер (приймає Telegram-повідомлення, керує TMA, стейт)
- **n8n** = фонова оркестрація (AI-виклики, нотифікації, звіти, ескалації)
- **Зв'язок:** aiogram → n8n через внутрішній webhook `http://localhost:5678/webhook/...`

### 1.3. Структура бази даних

| Таблиця | Призначення |
|---|---|
| `stores` | Магазини (name, address, coordinates, working_hours) |
| `users` | Усі користувачі системи (full_name, telegram_id, phone) |
| `roles` | Довідник ролей (name, can_assign_to, permissions) |
| `user_roles` | Зв'язка користувач ↔ роль (many-to-many) |
| `user_stores` | Прив'язка працівників до магазинів |
| `technicians` | Профіль техніка (specialization, rating, wallet_balance) |
| `ticket_categories` | Категорії поломок (Холод, IT, Електрика) з default_priority |
| `tickets` | Тікети (store_id, tech_id, status, type, category_id, description, review_text, rating_stars, ai_sentiment_score, ai_price, assigned_by, target_date, media_urls) |
| `ticket_history` | Аудит-лог змін статусів (ticket_id, changed_by, old_status, new_status, changed_at) |
| `transactions` | Монетні транзакції (tech_id, ticket_id, amount, type) |
| `user_sessions` | Стан сесії користувача (замість Redis) |

**Примітки до схеми:**
- Фото/голос до тікетів зберігаються в Supabase Storage, а посилання — в полі `media_urls (jsonb)` таблиці `tickets` (масив URL-ів)
- `ticket_history` — обов'язковий аудит-лог для бізнесу (хто, коли, що змінив)
- `target_date` у tickets — дата виконання (сьогодні або завтра, для задач від керівника)
- `assigned_by` у tickets — хто поставив задачу (для ієрархії)
- Мультитенантність поки не потрібна (1 клієнт), але при масштабуванні додається `tenant_id`

### 1.4. Безпека та автентифікація

- Валідація `initData` (HMAC-SHA256) на бекенді при кожній сесії TMA
- Row-Level Security (RLS) у Supabase для кожної таблиці
- Аудит-лог через `ticket_history`
- Не виставляти `service_role` ключ на клієнт
- Один Telegram-бот для всіх ролей — розподіл інтерфейсу по ролі користувача

## 2. ІНТЕРФЕЙСИ КОРИСТУВАЧІВ

### 2.1. Інтерфейс Техніка — Квест-борд (TMA)

**А. Хедер:**
- 💰 Золотий Баланс (поточний)
- 📊 Зароблено за цей місяць: X 💰
- 🔥 Згоріло минулого місяця: Y 💰 (незароблені монети з бюджету — різниця між виділеним бюджетом і фактично заробленим усіма техніками)
- ПІБ та рівень майстерності

**Б. Робочі зони (Accordion):**
1. 🚨 **КРИТИЧНІ ЗАВДАННЯ** — обов'язкові аварії, 0💰, виділені червоним
2. 👤 **ЗАВДАННЯ ВІД КЕРІВНИЦТВА** — завдання від керівника або заступника (голосові/текстові), 0💰. Показується: хто поставив, дата (сьогодні/завтра), опис
3. 📦 **ЗАГАЛЬНОДОСТУПНІ ЗАДАЧІ** — "ринок" додаткового заробітку, вартість у 💰. Розгортаються: Магазин → Список задач → Вартість

**В. Голосовий звіт:**
Технік може надиктувати звіт по виконаній задачі голосом. AI-Whisper перетворює на структурований текст, який зберігається як коментар до тікету.

### 2.2. Інтерфейс Магазину — Dashboard (TMA)

- **Кнопка SOS:** Швидка подача заявки (обов'язкове фото + Голос/Текст + вибір відділу)
- **Моніторинг:** Активні заявки + ПІБ техніка
- **Підтвердження виконання:** Працівник магазину підтверджує, що ремонт виконаний (це і є "закриття" тікету)
- **Відповідь на уточнення:** Якщо технік запитує уточнення — працівник отримує повідомлення і може відповісти текстом/голосом/фото
- **Зворотний зв'язок:** Оцінка якості після підтвердження
- **Автоповідомлення:** "Заявка прийнята → Технік у дорозі → Прибув → Очікує підтвердження"

### 2.3. Інтерфейс Керівника / Заступника (TMA + Веб)

**У TMA (мобільний, оперативний):**
- **Список технарів:** Вибір працівника зі списку
- **Кнопка "Поставити задачу":**
  1. Вибрати техніка (або заступника, для керівника)
  2. Надиктувати задачу голосом → AI-Whisper автоматично перетворює на структурований текст
  3. Обрати дату: "Сьогодні" або "Завтра"
  4. Підтвердити → задача з'являється у Квест-борді техніка в секції 👤
- **Моніторинг:** Статуси всіх активних задач

**У Веб-порталі (десктоп, аналітика):**
- Діаграми ефективності (Pie, Bar, Line Charts)
- AI-Sentiment Score по технарях
- Heatmap "Здоров'я мережі"

### 2.4. Інтерфейс Біг Боса (Веб-портал)

- Місячний бюджет монет + курс монети (1💰 = X грн)
- AI Pricing: право вето на вартість задач
- Burn-up Chart: контроль витрачання бюджету
- ROI-аналіз: швидкість закриття заявок vs витрати
- Алерти бюджету: 70% → 🟡, 90% → 🟠, 100% → 🔴 (платні квести заморожені)

## 3. AI-ФУНКЦІОНАЛ

### 3.1. AI-Консьєрж (Self-Service)

Простий FAQ-бот. Перед створенням тікету пропонує працівнику магазину варіанти самостійного вирішення:
- "Перевірте, чи підключено живлення"
- "Спробуйте перезавантажити пристрій"
- Якщо не допомогло → створення тікету

### 3.2. AI-Класифікатор

Автоматично визначає категорію поломки (Холод, IT, Електрика тощо) та пріоритет на основі опису.

**Відмовостійкість:**
- Retry 3 рази з exponential backoff
- Fallback: якщо OpenAI недоступний → тікет = `unclassified`, `manual_review: true`
- Логування кожного AI-рішення

### 3.3. AI-Оцінювач вартості (Structured Pricing)

Замість примітивного "оціни від 1 до 10":
- Промпт отримує контекст: категорія, опис, кількість рецидивів, середній час виконання
- AI повертає JSON з факторами: діагностика (1-3), запчастини (0-2), трудомісткість (1-3), терміновість (1-2)
- Фінальна ціна = сума факторів × коефіцієнт Біг Боса
- Після виконання — калібрація: порівняння AI-оцінки з реальною складністю
- Право вето у Біг Боса / Керівника

### 3.4. AI-Whisper (Голосові звіти та голосові задачі)

Три сценарії використання:
1. **Технік → Звіт:** Надиктовує звіт по ремонту → AI перетворює на структурований текст у БД
2. **Технік → Відповідь:** Звітує голосом по задачі від керівника → AI структурує і зберігає як відповідь до тікету
3. **Керівник/Заступник → Задача:** Надиктовує задачу для техніка → AI перетворює на опис задачі

Усі — через OpenAI Whisper API → STT → структурування тексту через GPT.

### 3.8. Уточнення проблеми (Clarification Request)

Технік, переглядаючи тікет, може натиснути кнопку **"Уточнити проблему"**:
1. Технік вводить питання текстом або голосом
2. Працівник магазину отримує push-повідомлення з питанням
3. Працівник відповідає (текст/голос/додаткове фото)
4. Відповідь додається до тікету, технік отримує нотифікацію
5. Статус тікету тимчасово змінюється на `awaiting_clarification` → повертається до `in_progress` після відповіді

### 3.5. Детектор рецидивів

AI автоматично попереджає техніка, якщо аналогічна проблема в цьому магазині виникала нещодавно. Підтягує інформацію з минулих тікетів.

### 3.6. AI-Історія ремонтів

Технік запитує: "Хто і коли ремонтував тут раніше?" → AI аналізує минулі тікети цього магазину та видає вижимку: ПІБ майстра, дату, суть робіт.

### 3.7. Sentiment Analysis

Аналізує текст відгуків магазину, щоб відрізнити технічні помилки техніка від суб'єктивного ставлення персоналу. Результат → `ai_sentiment_score`.

## 4. ГЕЙМІФІКАЦІЯ ТА ЕКОНОМІКА

### 4.1. Економіка "Duty First"

Технік може додавати платні квести (📦), але кнопка "Виконано" (нарахування монет) активується лише після закриття всіх 🚨 та 👤 задач на поточний день.

### 4.2. Прогресивна штрафна система

Від'ємного балансу НЕ буває. Мінімальний баланс = 0.
Штрафний дедлайн: **23:00** — невиконані обов'язкові задачі після цього часу = порушення.

| Порушення за місяць | Реакція |
|---|---|
| 1-ше | ⚡ Попередження + нотифікація керівнику |
| 2-ге | 🟡 -20% заробітку наступного дня |
| 3-тє | 🔴 Штраф 50 монет + особиста бесіда |
| 4+ | ⛔ Блокування платних квестів на тиждень |

### 4.3a. Логіка "Згоріло монет"

Відображається в хедері техніка та у звітах Біг Боса:
- **Бюджет місяця** = загальна кількість монет, виділена Біг Босом (напр. 10 000 💰)
- **Зароблено** = сума монет, фактично зароблених усіма техніками (напр. 2 000 💰)
- **Згоріло** = бюджет − зароблено (напр. 8 000 💰)
- Це метрика ефективності: чим менше "згоріло" — тим активніше техніки беруть платні квести

**Статуси тікету (підтверджені):**
`new` → `in_progress` → `awaiting_clarification` → `in_progress` → `pending_confirmation` → `completed` → `rated`

### 4.3. Гейміфікаційні механіки

- **Combo-магазин:** +15% бонусних монет за закриття 2+ квестів в одному магазині
- **Чистий сектор:** Тижневий бонус для команди за відсутність прострочених заявок
- **Leaderboard:** ТОП-10 тижня + персональний прогрес ("ти закрив на 15% більше") + різні номінації ("Найшвидший відклик", "Найвищий рейтинг")
- **Командний рейтинг:** Конкуренція між зонами/районами

### 4.4. Верифікація виконання

Двоетапна верифікація:
1. **Технік:** GPS-чекінг (радіус 50-100м від магазину) + обов'язкове фото "після"
2. **Працівник магазину:** Підтверджує виконання в TMA → тільки після цього тікет закритий

AI-аномалія-детектор: якщо технік "відвідав" 5 магазинів за 30 хвилин → автоматичний alert керівнику.

## 5. ЛОГІСТИКА ТА МАРШРУТИЗАЦІЯ

### 5.1. Денний план

Технік набирає пакет задач (🚨 + 👤 + 📦) на день.

### 5.2. Побудова маршруту (OpenRouteService)

1. Технік натискає "Побудувати маршрут"
2. Python-скрипт витягує координати всіх активних задач
3. **OpenRouteService API** (безкоштовний, 2000 запитів/день) оптимізує порядок відвідування, пріоритет 🚨 першим
4. Оптимізований маршрут відправляється як посилання на Google Maps / навігатор

> OpenRouteService — безкоштовний open-source сервіс на основі OpenStreetMap. Має optimization endpoint (задача комівояжера). Не потребує self-hosting. Реєстрація → API key → готово.

### 5.3. AI-підказки

Система пропонує взяти додаткові квести (📦), якщо вони знаходяться поруч із поточним маршрутом.

## 6. СИСТЕМА НОТИФІКАЦІЙ

Нотифікації реалізуються через Telegram Bot API (повідомлення в чат) та Supabase Realtime (оновлення TMA). Без окремої таблиці — nотифікації надсилаються напряму.

| Подія | Отримувач | Спосіб |
|---|---|---|
| Новий критичний тікет | Всі техніки | Telegram push |
| Тікет прийнято техніком | Магазин | Telegram msg |
| Технік у дорозі | Магазин | TMA Realtime |
| Задача від керівника | Технік | Telegram push |
| Тікет на підтвердженні | Магазин | Telegram msg |
| Тікет закрито | Магазин | Prompt для оцінки |
| Негативний відгук | Керівник + технік | Telegram msg |
| Бюджет 70/90/100% | Біг Босс | Telegram push |
| Штраф/попередження | Технік + керівник | Telegram msg |

## 8. ГОЛОСОВІ ЗАДАЧІ ВІД КЕРІВНИЦТВА

### Сценарій (детально):

1. Керівник (або заступник) відкриває TMA
2. Бачить список технарів (або заступників, для керівника)
3. Обирає працівника
4. Тисне кнопку **"Поставити задачу"**
5. Обирає дату: **"Сьогодні"** або **"Завтра"**
6. **Надиктовує задачу голосом** (або вводить текстом)
7. AI-Whisper перетворює голос → структурований текст
8. Керівник бачить preview тексту → підтверджує або перезаписує
9. Задача зберігається в `tickets` з:
   - `type = 'manager'`
   - `assigned_by = user_id керівника/заступника`
   - `tech_id = обраний технік`
   - `target_date = сьогодні або завтра`
10. Технік отримує push-нотифікацію: "Нова задача від [ПІБ]: ..."
11. Задача з'являється у його Квест-борді в секції 👤

### Правила ієрархії:
- Керівник може ставити задачі → технікам + заступникам
- Заступник може ставити задачі → лише технікам
- Працівник магазину НЕ може ставити задачі (тільки створює тікети через SOS)
- Біг Босс може все, але зазвичай працює через веб-портал

## 12. РЕЄСТРАЦІЯ ТА ОНБОРДИНГ

### 12.1. Магазини

**Початкове завантаження:** CSV-імпорт через веб-портал (адмін-панель).
Формат CSV: `назва, адреса, координати (lat/lng), робочі години`

**Новий магазин:** Форма "Додати магазин" у веб-порталі керівника.

### 12.2. Техніки (самореєстрація з підтвердженням)

1. Технік натискає `/start` у боті
2. Бот питає: "Оберіть вашу роль" → технік обирає "Технік"
3. Бот просить ввести: ПІБ, спеціалізацію (Холод / IT / Електрика / Універсал)
4. Заявка потрапляє до **керівника** → push-повідомлення "Новий технік: Іванов І.І., Холод"
5. Керівник натискає ✅ Підтвердити або ❌ Відхилити
6. Після підтвердження → технік отримує повідомлення "Вас додано до системи" → бачить Квест-борд

### 12.3. Працівники магазинів (самореєстрація з підтвердженням)

1. Працівник натискає `/start` у боті
2. Обирає роль "Працівник магазину"
3. Бот показує список магазинів → працівник обирає свій
4. Вводить ПІБ
5. Заявка до **керівника** на підтвердження
6. Після підтвердження → працівник бачить Dashboard магазину (SOS-кнопка)

### 12.4. Заступники (самореєстрація з підтвердженням)

1. Натискає `/start` → обирає "Заступник керівника"
2. Вводить ПІБ
3. Заявка до **керівника** на підтвердження
4. Після підтвердження → бачить інтерфейс заступника

### 12.5. Керівник та Біг Босс

Створюються вручну в БД при первинному налаштуванні системи (seed). Це разова дія.

### Захист від стороннього доступу:
- Без підтвердження керівника ніхто не отримує доступ
- Бот відповідає незареєстрованим: "Ваша заявка на розгляді у керівника"
- Керівник бачить список очікуючих заявок у TMA

## 14. МІНІМАЛЬНІ ПІДСИЛЕННЯ MVP + ФІШКИ (БЕЗ УСКЛАДНЕННЯ)

> Цей розділ **не ламає основну структуру**. Це “підсилення” та “фішки”, які додають надійності, безпеки й кращого UX, але не змінюють базові сценарії (SOS → робота техніка → підтвердження → оцінка).

### 14.1. 5 ключових підсилень (рекомендовано зробити зразу)

1) **Журнал подій / доставка повідомлень (щоб нічого не губилось)**
- Суть: система веде простий “журнал”: що сталося (новий тікет, зміна статусу, AI-аналіз, відправка нотифікації) і чи це “успішно/помилка”.
- Навіщо: якщо щось не дійшло або продублювалось — видно **де зламалось** і можна **повторити** без хаосу.
- Мінімальна реалізація: одна таблиця типу `events_outbox` і/або `notification_log` + дедуп-ключ, щоб повторні події не ламали дані.

2) **Медійка (фото/голос) “в сейфі” + тимчасові посилання**
- Суть: фото/голос зберігаються приватно, а користувачі отримують **тимчасовий доступ** (signed URL) на перегляд.
- Навіщо: менше ризику випадкового витоку фото (це часто найболючіше для бізнесу).
- Мінімальна реалізація: приватні buckets у Supabase Storage + видача signed URL на клієнт/бот.

3) **Монети “як банківська виписка” (ledger)**
- Суть: “баланс” — це не число, яке хтось колись переписав; баланс = сума транзакцій у `transactions`.
- Навіщо: майже неможливо “зламати” баланс, простіше робити звіти/аудит і розбирати спірні ситуації.
- Мінімальна реалізація: всі нарахування/штрафи — тільки як записи в `transactions`; баланс або рахується запитом, або кешується тригером.

4) **Справедливі штрафи: прості “блокери” без зміни процесу**
- Суть: якщо задача не закрита не з вини техніка — це має бути видно.
- Додати 2 прості причини:
  - **`blocked_by_store`** — чекаємо відповідь/підтвердження магазину
  - **`blocked_by_parts`** — чекаємо запчастини
- Навіщо: менше токсичності, штрафи стають “чесними”, керівник бачить реальні причини затримок.
- Мінімальна реалізація: поле `block_reason` у `tickets` (або простий статус/підстатус).

5) **GPS — як підказка, а не “замок”**
- Суть: GPS-чекінг корисний, але інколи помиляється.
- Правило: якщо GPS не пройшов — не блокуємо роботу; fallback = “фото після” + підтвердження магазину.
- Навіщо: менше ситуацій “я на місці, а система не дає закрити”.
- Мінімальна реалізація: логувати результат GPS як сигнал, але не робити його єдиною умовою закриття.

### 14.2. 4 прості фішки з “вау-ефектом” (додати без зайвих екранів)

A) **SOS-шаблони (швидкий старт заявки)**
- 3–5 кнопок типових проблем: “Не холодить”, “Каса/IT”, “Електрика”, “Інше”.
- Після вибору — фото + голос/текст.
- Ефект: швидше і точніше створення тікетів, менше “води” в описі.

B) **Рядок “Що мене блокує?” у техніка**
- В квест-борді показувати одним рядком: що саме заважає взяти платні квести (напр. “не закриті 🚨 за сьогодні”).
- Ефект: прозорість → менше негативу до “Duty First”.

C) **“Поруч із маршрутом” — показувати лише ТОП-3 квести**
- Не грузити техніка списками: показати 3 найвигідніші поруч (вигода/час).
- Ефект: більше виконаних платних квестів без зайвої метушні.

D) **Низька оцінка = коротка причина**
- Якщо 1–2 зірки — просимо вибрати причину (повільно/якість/комунікація/інше 1 фраза).
- Ефект: менше “несправедливих” оцінок, краща аналітика і керованість якості.

### 14.2a. Додаткові прості фішки (опційно)

E) **“Тихий режим” для магазину (анти-спам)**
- Суть: якщо статуси часто повторюються (типу “у дорозі/прибув”), бот не шле 5 повідомлень підряд, а зливає в 1 коротке.
- Ефект: персонал магазину не “вигорає” від спаму, але контроль зберігається.

F) **Авто-нагадування магазину про підтвердження**
- Суть: якщо тікет у `pending_confirmation` довше, ніж X годин — м’який пінг + кнопки “Підтвердити” / “Потрібно доробити”.
- Ефект: тікети не “висять”, закриття швидше, менше ручних дзвінків.

G) **Міні-чеклист “після ремонту” (3 пункти)**
- Суть: при завершенні технік відмічає 3 речі: “що зроблено”, “фото після”, “чи працює зараз”.
- Ефект: менше конфліктів і переробок, краща якість звітів без довгих форм.

H) **Обмеження на “Побудувати маршрут” + кеш**
- Суть: кнопку можна натискати не частіше ніж раз на 5 хв (і/або показувати попередній результат), перерахунок лише якщо змінився набір задач.
- Ефект: економія лімітів OpenRouteService, менше зайвих натискань, стабільніший UX.

### 14.3. Основні ризики простими словами + як їх закриваємо

1) **“Повідомлення можуть загубитись або прийти двічі”**
- Чому: інтернет/Telegram/n8n/сервер можуть перезапускатись; webhooks інколи повторюються.
- Рішення: **журнал подій/нотифікацій** + **dedupe** (повторна подія не створює дубль).

2) **“n8n впав — і фонові дії не виконались”**
- Чому: n8n локальний процес, може зупинитися або зависнути.
- Рішення: події не “летять в повітря”, а записуються в **outbox**; n8n/бот підбирає їх після відновлення.

3) **“Фото/голос можуть стати публічними випадково”**
- Чому: публічні URL або неправильні права на storage.
- Рішення: приватне зберігання + **тимчасові** (signed) посилання.

4) **“Баланс монет може роз’їхатись”**
- Чому: якщо баланс — просто поле, яке оновлюють різні частини системи, з’являються помилки/гонки.
- Рішення: **ledger** через `transactions`: усе — транзакціями, баланс — похідний.

5) **“GPS інколи бреше — і технік не може закрити задачу”**
- Чому: геолокація нестабільна в приміщеннях/щільній забудові.
- Рішення: GPS як сигнал + fallback “фото після” + підтвердження магазину.

6) **“Штраф прилетів несправедливо (магазин не відповів / запчастин нема)”**
- Чому: система бачить лише “не закрито”, але не бачить причину.
- Рішення: прості блокери `blocked_by_store` / `blocked_by_parts` + окреме правило, що такі блокери не рахуються як провина техніка.

7) **“Маршрутизація може впертись у ліміти ORS”**
- Чому: часті перерахунки маршруту на багатьох техніків.
- Рішення: кеш маршруту на 10–15 хв + перерахунок тільки коли змінився набір задач.
